# 99 → Vocode: Concepts, Patterns, and Boundaries

Analysis of **ThePrimeagen/99** (vendored under `third-party-research/99/`) as design inspiration for Vocode. **Part I** (§§1–11) paraphrases 99’s architecture and maps it to Vocode conceptually—no third-party code. **Part II** (§§12–16) is an **implementation-oriented specification**: typed edit protocol, target resolution, end-to-end pipelines, system guarantees, and failure handling.

---

## 1. Core philosophy of 99

**Problem it targets**  
99 is a Neovim-native layer that connects the editor to external AI CLI backends so the human stays in control of *where* work happens while still using model strength for search, explanation, and localized rewrites.

**Stated intent**  
The project frames itself as **augmentation, not replacement**: keep “hand coding” central; use LLMs to surface information and assist with focused edits rather than owning the whole workflow.

**How it differs from typical AI coding tools**  
- **Editor-local orchestration**: Requests are built from buffer state (selection, path, optional markdown “agent” files), then handed to a **pluggable provider** that shells out to tools like OpenCode, Claude Code, Cursor Agent, or Gemini CLI—not a single vendor-locked chat panel.  
- **Operation modes are explicit**: “visual” (selection replace), “search” (project-wide locate), “vibe” (broader agentic session with different output contract), “tutorial”, plus an extensions **Worker** that combines a saved work description with search.  
- **Directional emphasis (per README)**: Early focus was replacing selected code; the author’s current emphasis is **`search` and work-tracking**—finding relevant locations before (or instead of) large automated edits.

**Contrast with Vocode’s positioning**  
99 still leans on **general-purpose agent CLIs** for execution; Vocode aims for a **voice-first, scoped edit engine** with stronger predictability. 99’s **visual** path is the closest cousin to Vocode’s “edit only this range” philosophy; **vibe** is closer to conventional agent coding and is a weaker match for Vocode’s non-destructive defaults.

## 2. Key workflows in 99

### Visual selection editing

1. User enters **visual mode** and selects a region (recommended so the “last visual selection” is unambiguous).  
2. Invoking **visual** captures the selection’s **geometry** (buffer, start/end), normalizes edge cases (e.g. line-mode selection), and drops **editor marks** bracketing the range so the span can still be found after the selection is cleared.  
3. User provides a **natural-language instruction** (floating prompt buffer or programmatic `additional_prompt`).  
4. The system builds a **structured prompt**: selection metadata, selected text, and often the **entire buffer** as surrounding context—so the model sees local intent plus file-level context.  
5. The provider runs; on success, the **response text replaces only the marked range** (empty response aborts). Marks must remain valid or the edit is refused.

**Implication**: “Where to edit” is fixed **before** the model runs; the model’s job is to produce **replacement buffer text** for that span.

### Search-based workflows

1. User opens **search** (prompt buffer or `additional_prompt`).  
2. A **search-specific system prompt** instructs the model to return **only** a machine-parsable list of locations (absolute paths, line/column, highlight span, single-line notes).  
3. The response is **parsed into a list of locations**; if non-empty, results open in the **quickfix list** for navigation.

**Implication**: Search is **non-destructive navigation**—it does not apply edits. The human (or a future automation step) decides what to open or change next.

### Context injection (`@files`, `#rules`, markdown agents)

- **`@…` and `#…` tokens** in the user prompt are resolved via a **completion provider registry**: the prompt string is scanned for tokens, each provider validates and **expands to reference content** appended to the outgoing context.  
- **`#rules`** map to discovered rule/skill files (e.g. directories of `SKILL.md`-style content).  
- **`@files`** map to project files (discovery respects git-aware file lists and size/exclude limits when configured).  
- **`md_files` (e.g. `AGENT.md`)**: On finalize, the system walks **upward from the buffer’s directory toward the project root** and **prepends** any matching filenames found along the path—automatic, path-based “agent instructions” without typing `@`.

**Optional API hook**: `additional_rules` lets keymaps or scripts **inject rule bodies** without embedding them in the typed prompt.

### Request lifecycle (end-to-end)

1. **Operation entry** (`visual` / `search` / `vibe` / …): construct a **Prompt context** (trace id, logger, tmp paths, operation type).  
2. **User text**: from `additional_prompt` or from a **capture window** (with submit keymap); capture path can merge **named rules** referenced in the buffer into `additional_rules`.  
3. **Prompt assembly**: operation template + user text wrapped in labeled sections; **references** from `@`/`#`/`additional_rules` concatenated into the agent context.  
4. **Finalize**: ensure temp prompt/response files exist; inject **ancestor markdown** files; for visual ops, inject **file location** and **range text** again; inject **temp file location** and, for several modes, **strict instructions** that the agent must route output through the temp file and not chatty alternatives.  
5. **Provider**: spawn CLI; stream stdout/stderr to optional UI; on exit, **read the temp file** as the canonical result.  
6. **Completion handler**: **visual** → replace marked range; **search** → parse → quickfix; cancel/kill paths clean up.  
7. **Tracking**: successful requests can be **serialized** for later reopen (search/vibe reopen quickfix; visual reopen is noted as incomplete in API).

## 3. The edit model in 99

### How edits are scoped

- **Primary scoped edit**: **Visual** — scope is the **user’s selection**, reified as a **range with stable marks**.  
- **Search / vibe**: **No direct buffer replacement** in the core search op—output is **locations** (and for vibe, a different contract oriented around a temp transcript of actions).  
- **Implicit wide context**: Even for visual, the prompt may include the **full file**, so cognitive scope for the *model* is large even though the **applied edit** is range-limited.

### How ranges / selections are used

- Ranges are **first-class** (buffer id, start/end positions), with helpers to **read text**, **replace text**, and derive from **visual selection** or **Tree-sitter nodes** (for potential non-visual flows).  
- **Marks** anchor the endpoints so asynchronous provider completion can still find the span.

### How changes are applied

- **Visual**: **Synchronous buffer text splice** over the range derived from marks; guardrails if marks were invalidated or response empty.  
- **Provider side**: Commands are built per backend; **success is tied to reading a temp file** after the process exits—so the integration assumes the tool **writes** deliverables to disk, not only stdout.

**For Vocode implementers**: 99’s **visual** path matches a **`replace_range`-style** edit, formalized in **§12**; **search** matches a **location list** that should feed **target resolution** (**§13**) and the **search phase** of the **search → act** model (**§4**), not the edit engine directly.

## 4. Search as a first-class primitive

**Mechanics**  
Search uses a **dedicated prompt template** that defines a **rigid line format** (path, line, column, line-span, notes). A small parser turns each line into a **location record** for the UI.

**Why it matters**

- **Separation of concerns**: “Find candidates” is a distinct phase from “mutate buffer.”  
- **Human-in-the-loop navigation**: Quickfix-style lists match how developers already triage many hits.  
- **Composable with higher-level workflows**: The **Worker** extension builds a **work description**, then runs **search** with a checklist-style prompt (diff, tests, etc.) to **surface what remains** for that work item.

**Vocode angle**: Treat **structured search results** as a **first-class output type** from intent parsing (voice: “find all usages of X”) before any apply step.

### Formal model: search phase vs edit phase (“search → act”)

Vocode should treat **search** and **edit** as **distinct pipeline stages** with an explicit handoff:

| Phase | Purpose | Produces | May mutate workspace? |
|--------|---------|----------|------------------------|
| **Search** | Turn a (possibly vague) question into **candidate locations** | `SearchResult[]` (see §13) with stable ids | No |
| **Target resolution** | Choose **zero or one** primary edit target from selection, symbol, search result, or clarification | `ResolvedTarget` | No |
| **Edit** | Apply a **validated** `EditBatch` (§12) | Editor transactions | Yes, only as specified |

**Why edits must not run directly from vague intent**  
Unscoped natural language (“clean this up”, “fix the bug”) does not fix **where** changes belong. Applying an LLM’s guess without a resolved target invites **whole-file drift**, wrong symbol edits, and silent collateral changes. Vocode’s default path is: **intent → (optional search) → explicit target → scoped LLM call → typed edits**.

**How search results feed the edit engine**  
Search output is **not** pasted into the buffer. It is stored as **structured hits** (uri, range, label, optional snippet). A subsequent command (“edit the second hit”, “refactor this match”) sets **`SearchResultTarget`** (or narrows to a **`SelectionTarget`** after the user moves focus). The LLM for the edit step receives **only** the resolved scope (range + minimal surrounding context), not the entire search response as free-form instructions to “change the project.”

**Composition rule**  
Any workflow that feels like “search then edit” should be implemented as **two user-visible commands** (or one compound command that **internally** runs search, presents results, then requires **confirmation or disambiguation** before invoking §12). Skipping target resolution is an **explicit escape hatch**, not the default.

## 5. Context injection model

**Layers (conceptual order)**

1. **System / role fragments** (generic assistant framing).  
2. **Operation template** (visual vs search vs vibe vs tutorial)—defines task and output rules.  
3. **User prompt** wrapped in labeled XML-like blocks for clarity.  
4. **Explicit references**: expanded content from `@` and `#` tokens in the user string.  
5. **Programmatic rules**: optional extra rule documents from the API.  
6. **Path-walk markdown**: `AGENT.md`-style files from current file’s directory up to workspace root.  
7. **Execution constraints**: temp file path, and for several modes, **hard rules** about where the model may write results.

**Why results improve**

- The model gets **stable structure** (sections, rules, examples) repeated every request.  
- **@ / #** make “pull in this file or rule” **explicit and inspectable** before send.  
- **Ancestor markdown** automatically grounds requests in **repo-local conventions** without the user repeating them.

**Vocode mapping**: Reuse **layered context** with voice—e.g. spoken “add the API guidelines” resolving to the same **reference slots** as `#rules` today.

## 6. Abstractions and concepts (editor-agnostic)

| Concept | Description |
|--------|-------------|
| **Request context object** | Holds operation type, lifecycle state, trace id, tmp artifact paths, logging, and post-success data (e.g. location list). |
| **Operation-specific prompt templates** | Different system instructions per task; same assembly pipeline. |
| **Range-based editing** | Immutable-ish span + **replace** operation; anchoring for async completion. |
| **Reference resolution pipeline** | Parse user text → resolve tokens → append structured context chunks. |
| **Provider boundary** | Swap backends; common contract: **full prompt string in**, **artifact on disk out**, optional streaming. |
| **Cancellation** | Kill subprocess; discard partial results; clean marks/UI. |
| **Tracking / history** | Serialize successful requests for revisit (implementation varies by operation). |
| **Search → act** | Emit **typed locations**; UI or automation consumes them. |
| **Scripted intents** | `additional_prompt` on keymaps for fixed instructions (e.g. “run tests and list failure locations”). |

## 7. What is Neovim-specific (do not carry over)

- **Visual mode**, `'<`/`'>`, and **mark** APIs for anchoring.  
- **Quickfix** (`setqflist`, `copen`) as the search result surface.  
- **Floating/split windows** for prompt capture and tutorials.  
- **`vim.system`**, `vim.schedule`, buffer text APIs, keymaps, file path helpers.  
- **Telescope / fzf-lua / nvim-cmp / blink** extension UIs for model/provider pickers and completions.  
- **Plugin lifecycle** tied to Neovim’s startup and buffer-local filetypes.

Vocode on VS Code (or elsewhere) should reimplement **equivalents** (Problems panel or peek list, selection APIs, task cancellation), not these APIs.

## 8. Mapping to Vocode

| 99 concept | Vocode equivalent | Intent parsing | Target resolution | Edit engine | Voice notes |
|------------|-------------------|----------------|-------------------|-------------|-------------|

| Search op | **Locate** / “find candidates” | “Find …”, “where is …” → search intent | Results list, filter by file/glob | No apply until confirmed | Read back hit count + first labels; “open third match” |

## 9. What Vocode should borrow

1. **Range-anchored async edits** with **validation** before apply (marks invalid → abort; empty → abort).  
2. **Separate search primitive** with a **strict structured output** format and a **typed parse** into locations—then **navigate** before editing.  
3. **Layered prompt assembly**: operation template + labeled user context + resolved references + workspace rules.  
4. **`@`-style file and `#`-style rule inclusion** with a **deterministic expansion** step (auditable context).  
5. **Ancestor-directory markdown** (or equivalent) for **auto-loaded project conventions**.  
6. **Scripted `additional_prompt`-style commands** for repeatable voice or key-bound flows.  
7. **Worker pattern**: persist a **work description** + run **gap-finding search** against diffs/tests—useful for structured iteration.  
8. **Explicit cancel** of long-running provider work—critical when speech triggers heavy jobs.  
9. **Trace / request id per operation** for logging and support (99 attaches loggers per request).

---

## 10. What Vocode should not copy

1. **Neovim-only coupling** (marks, quickfix, `vim.system`)—re-express as **editor SDK** operations.  
2. **Treating external agent CLIs as the only edit path** without a **Vocode edit contract** (diff hunks, range patches, symbol ops)—99’s **visual** is good; unconstrained multi-file agent behavior is **not** Vocode’s default safety model.  
3. **Sending the entire file as context on every visual edit** without **token/safety policy**—Vocode should prefer **minimal sufficient context** when voice-driven.  
4. **Over-reliance on temp-file transcripts** as the user-visible result—voice users need **clear confirmation** and **undo granularity**.  
5. **Implicit “last visual selection” pitfalls**—voice should **resolve scope explicitly** (selection → symbol → file region) rather than hidden editor state.  
6. **Beta API volatility** as a model—Vocode should version **intent and patch schemas** for extensions.

---

## 11. Gaps / opportunities (voice + scoped editing)

**What Vocode can do better**

- **Spoken target disambiguation**: After search, resolve “first / second / in `bar.ts`” without manual quickfix motion.  
- **Smaller LLM payloads**: Voice latency rewards **tight scopes**—avoid whole-file context when a symbol range suffices.  
- **Deterministic apply path**: Parse model output into **patches**, run **static checks**, then apply—reducing “paste surprise.”  
- **Undo as a product feature**: One utterance → **one undo transaction** across possibly multiple small edits.  
- **Hands-free iteration**: “Redo with constraint X” on the **same range** without re-selecting.  
- **Search without typing**: Same **structured location format** as 99, but **spoken queries** and **audio summaries** of hits (accessibility + flow).  
- **Governed agent mode**: If a “vibe-like” mode exists, require **explicit voice confirmation** before any non-scoped or multi-file write.

## 12. Proposed Vocode Edit Protocol

### 12.1 Shared primitives

```ts
/** 0-based half-open line/character, UTF-16 code units (align with LSP). */
interface Position {
  line: number;
  character: number;
}

/** Inclusive start, exclusive end. */
interface Range {
  start: Position;
  end: Position;
}

/** Stable document identity for validation. */
interface DocumentAnchor {
  uri: string;
  /** Monotonic per-document version from the editor (e.g. LSP version). */
  version: number;
}

/** Optional integrity check: if present, must match exact buffer text at range. */
interface RangeFingerprint {
  /** Literal UTF-8 (or normalized newline) slice of document text at range at capture time. */
  expectedText: string;
}
```

### 12.2 Edit action union

All actions are **immutable data** produced only after LLM output is parsed and validated. The editor applies them inside a **single undo group** (§12.5).

```ts
type EditAction =
  | ReplaceRangeAction
  | InsertAtPositionAction
  | DeleteRangeAction
  | MultiLocationPatchAction;

interface ReplaceRangeAction {
  kind: "replace_range";
  document: DocumentAnchor;
  range: Range;
  newText: string;
  /** If set: current buffer slice must equal this or the action fails (stale / mismatch). */
  fingerprint?: RangeFingerprint;
}

interface InsertAtPositionAction {
  kind: "insert_at_position";
  document: DocumentAnchor;
  position: Position;
  text: string;
  /** Default: insert before the character at `position` (LSP-style). */
  placement?: "before" | "after";
}

interface DeleteRangeAction {
  kind: "delete_range";
  document: DocumentAnchor;
  range: Range;
  fingerprint?: RangeFingerprint;
}

interface MultiLocationPatchAction {
  kind: "multi_location_patch";
  /** Ordered list of atomic edits; applied in order after global validation. */
  children: Array<ReplaceRangeAction | InsertAtPositionAction | DeleteRangeAction>;
}
```

**Normative constraints**

- **`multi_location_patch` flattening**: Implementations may normalize nested `multi_location_patch` to a flat list; the protocol disallows nesting `multi_location_patch` inside `children` (reject at parse time).
- **No implicit path**: Every action requires an explicit `document.uri` (no “current file” wildcard in the persisted protocol; the orchestrator binds uri before apply).

### 12.3 Validation rules (pre-apply)

Run **after** JSON/schema validation, **immediately before** editor mutation, against **live** document state:

| Check | Applies to | Rule |
|--------|------------|------|
| Version match | All | `document.version` must equal the editor’s current version for that uri. |
| Range in bounds | `replace_range`, `delete_range`, fingerprinted ops | `range` lies within document; `start <= end` lexicographically. |
| Fingerprint | Ops with `fingerprint` | Text at `range` in the live document must equal `expectedText` (newline policy fixed per implementation). |
| Insert position | `insert_at_position` | `position` is valid (line exists; character at line end allowed per LSP rules). |
| Multi-batch consistency | `multi_location_patch` | Compute **non-overlapping** ranges after ordering; if overlaps exist, **reject entire batch** unless explicitly merged by a deterministic conflict resolver (default: reject). |
| File scope | All | Default policy: **reject** if any uri is outside workspace roots unless user granted elevated scope for this command. |

**Failure conditions (hard fail — do not apply)**  
- Version mismatch → **stale document**.  
- Fingerprint mismatch → **stale range** or model hallucinated range.  
- Overlapping patches in one batch → **invalid batch**.  
- Parse error, unknown `kind`, or extra fields forbidden by schema → **invalid protocol output**.

**Soft fail (implementation option)**  
- Single-file `replace_range` with no fingerprint: optionally allow apply with **user confirmation** only (voice: “buffer changed; say confirm to apply anyway”).

### 12.4 LLM output → `EditAction[]`

1. **Constrain the model**: Use **structured output** (JSON schema / tool call) whose payload is **`{ actions: EditAction[] }`** or a single action—no free-form “here’s the new file” as the primary path for default Vocode.  
2. **Parse and normalize**: Validate types, `kind` discriminant, uri format, range ordering.  
3. **Bind documents**: Map logical file references to workspace uris; reject ambiguous relative paths.  
4. **Capture fingerprints** (recommended): When the LLM request is built, attach **`expectedText`** for every range the model is allowed to edit (the exact current slice). The model returns ranges that must match; if the model returns different text, validation fails.  
5. **Version stamp**: Orchestrator sets `document.version` from the editor at **enqueue time**; right before apply, re-check version.  
6. **Emit `EditBatch`**: Wrap the validated list for undo grouping.

```ts
interface EditBatch {
  /** Correlates logs and voice session. */
  commandId: string;
  actions: EditAction[];
  /** If true, failure of any action rejects the whole batch (default true). */
  atomic?: boolean;
}
```

### 12.5 Undo grouping

- **One user command** (one voice utterance that triggers an edit, or one confirmed UI action) → **exactly one** editor `WorkspaceEdit` (or equivalent) submitted inside **one undo transaction**.  
- **`multi_location_patch`**: All `children` are part of **the same** undo transaction.  
- If the implementation applies actions sequentially for technical reasons, the editor must still expose **one undo** to the user (e.g. VS Code `WorkspaceEdit` batching or `editor.edit` with single undo stop).

### 12.6 Preventing invalid / unsafe edits

- **Stale ranges**: Prefer **version + fingerprint**; on failure, re-run target resolution or ask the user to re-issue the command.  
- **Mismatched content**: Fingerprint mismatch means the model targeted the wrong span—**do not** apply.  
- **Whole-file substitution**: Reject any single `replace_range` whose range spans the **entire document** unless `intent.allowFullFileReplace === true` (explicit user opt-in for that command).  
- **Deterministic transforms**: When the intent is mechanical (rename local, extract constant), prefer **AST / LSP refactor** or regex with verified pre-image over LLM-generated full replacement where feasible.

## 13. Target Resolution System

### 13.1 Target types

```ts
type ResolvedTarget =
  | SelectionTarget
  | SymbolTarget
  | FileTarget
  | SearchResultTarget;

interface SelectionTarget {
  kind: "selection";
  document: DocumentAnchor;
  range: Range;
  fingerprint: RangeFingerprint;
}

interface SymbolTarget {
  kind: "symbol";
  document: DocumentAnchor;
  /** From LSP or tree-sitter; includes which scope (name, kind). */
  symbolId: string;
  range: Range;
  /** Which syntactic span the user may edit (default for vague “this function”: `body`). */
  span: "name" | "signature" | "body" | "full_symbol";
}

interface FileTarget {
  kind: "file";
  document: DocumentAnchor;
  /** Optional sub-range when user said “this file” but caret was inside a known span. */
  focusRange?: Range;
}

interface SearchResultTarget {
  kind: "search_result";
  /** Id from the search phase result set (stable within session). */
  hitId: string;
  document: DocumentAnchor;
  range: Range;
  fingerprint?: RangeFingerprint;
}
```

### 13.2 Resolution priority (default)

When the user does not name a target explicitly, evaluate in order **stop at first unambiguous match**:

1. **Explicit selection** — non-empty editor selection in the active editor → `SelectionTarget`.  
2. **Explicitly referenced symbol** — utterance or prior slot contains a **unique** symbol reference (name + kind + file) resolvable via LSP/workspace index → `SymbolTarget`.  
3. **Inferred symbol at caret** — caret inside a documented symbol range (LSP `textDocument/documentSymbol` or equivalent) and intent implies “this function/class” → `SymbolTarget` with `span` from intent defaults (prefer **body** over **full_symbol** for edits).  
4. **Active search result** — user previously ran search; utterance references ordinal (“second result”) or label that maps to one hit → `SearchResultTarget`.  
5. **Fallback** — **clarification required** (§13.3); do not call the edit LLM.

**Overrides**  
- Voice or UI **slot filling** (“in `foo.ts`”, “the constructor”) can jump to **symbol** or **file** before step 1 if the phrase is **unambiguous**.  
- If intent is **search-only**, skip edit resolution entirely and run the search phase only.

### 13.3 Ambiguity and clarification (voice-critical)

| Situation | System behavior |
|-----------|-----------------|
| Multiple symbols match name | Spoken or UI disambiguation: list **at most 3** candidates with file + kind; ask user to pick. |
| Selection empty and caret in whitespace | Do not infer file-wide target; ask **what to edit** or offer **search**. |
| Ordinal out of range (“edit fifth”) but only three hits | Report invalid reference; offer to repeat search summary. |
| Search results stale (buffer changed) | Re-validate fingerprint or version; if invalid, say results are stale and offer **refresh search**. |

**When to ask (normative)**  
- Zero targets after priority walk → **must clarify**.  
- More than one plausible symbol without disambiguator → **must clarify**.  
- Edit intent + **no** resolved target → **must not** invoke edit LLM.

### 13.4 Output of the resolver

```ts
type TargetResolution =
  | { status: "ok"; target: ResolvedTarget }
  | { status: "clarify"; prompt: string; candidates?: ResolvedTarget[] }
  | { status: "search_only"; query: string };
```

This object is what downstream **LLM input construction** uses to bound context (send **only** `target.range` ± small margin, not whole files unless policy allows).

## 14. End-to-End Voice → Edit Flow

### Example A — User modifies existing code (scoped edit)

| Step | Artifact |
|------|----------|
| **1. Voice input** | “Extract the guard clause into a function called `ensureReady`.” |
| **2. Parsed intent** | `{ operation: "edit", technique: "extract_guard", newName: "ensureReady" }` — intent classifier + slot extraction. |
| **3. Target resolution** | Non-empty selection covering the guard block → `SelectionTarget` with `fingerprint` of selected text. |
| **4. LLM input construction** | System: edit protocol rules + JSON schema for `EditAction[]`. User: instruction + **only** selected lines + 2–5 lines context (not whole file unless policy says otherwise). Attach `document.version`, range, `expectedText`. |
| **5. LLM output** | `{ actions: [{ kind: "multi_location_patch", children: [ /* insert new function above */, /* replace_range on selection */ ] }] }` |
| **6. Edit actions** | Parse → validate version + fingerprints + non-overlap → `EditBatch` with `atomic: true`. |
| **7. Editor application** | Single `WorkspaceEdit` / batched transaction applying insert then replace (order preserved). |
| **8. Undo behavior** | One undo restores pre-extract state. |

### Example B — User searches, then edits

| Step | Artifact |
|------|----------|
| **1. Voice input** | “Find where we handle OAuth errors.” |
| **2. Parsed intent** | `{ operation: "search", topic: "OAuth error handling" }` — **no** edit actions. |
| **3. Target resolution** | N/A for edit; emit `TargetResolution` as `{ status: "search_only", query: "..." }` or run search pipeline directly. |
| **4. LLM input construction (search)** | Search-only prompt; structured output → list of `{ hitId, uri, range, note }`. |
| **5. LLM output (search)** | Parsed hits stored in **session search buffer**; surface via UI/TTS: count + short labels. |
| **6. Edit actions** | *None yet.* |
| **7. Editor application** | Optional: reveal first hit or show list; **no** automatic buffer mutation. |
| **8. Second voice input** | “Edit the second one to log the refresh token failure.” |
| **9. Parsed intent (edit)** | `{ operation: "edit", ordinal: 2, instruction: "log refresh token failure" }`. |
| **10. Target resolution** | `SearchResultTarget` from hit `2` → re-fetch live range + fingerprint from current document. |
| **11. LLM input (edit)** | Scoped prompt around that **range only** + instruction; include version + `expectedText`. |
| **12. LLM output** | `replace_range` or small `multi_location_patch` **only** inside that span (enforced by validator + max-range policy). |
| **13. Editor application** | Single undo group. |
| **14. Undo behavior** | One undo reverts that hit’s edit only; search results list unchanged. |

## 15. System Guarantees

Vocode **SHALL** uphold the following unless the user has **explicitly** enabled a documented override mode (e.g. “whole-file assist”):

1. **No unintended file-wide edits** — Default intents reject `replace_range` spanning full document; multi-file edits require explicit scope grant.  
2. **Scoped and reversible** — Every default-path mutation is expressible as §12 actions and is undone with **one user undo**.  
3. **One user command = one undo group** — A single interpreted voice command that results in applies produces **one** undo transaction (§12.5).  
4. **Validate before apply** — No buffer mutation until §12.3 checks pass (or explicit user override after stale warning).  
5. **Determinism preference** — For mechanical intents, the system **prefers** deterministic transforms (LSP, AST, fixed templates) and uses the LLM only when necessary; when the LLM is used, output must still pass **structural** validation.  
6. **Search/edit separation** — Vague locate requests do not mutate the workspace; they complete the **search phase** only (§4 formal model).

## 16. Failure Modes & Safeguards

| Failure | Detection | System response | User feedback (voice + UI) |
|---------|-----------|-----------------|----------------------------|
| **LLM returns invalid output** | JSON/schema parse error; unknown `kind`; overlapping ranges | Abort apply; log structured error | “I couldn’t use that response. Try again or narrow the request.” |
| **Range becomes stale** | `document.version` mismatch or `fingerprint` mismatch | Abort apply; discard batch | “The file changed. Say ‘retry’ or select the code again.” |
| **Ambiguous voice command** | Resolver returns multiple candidates or none | No edit LLM call; enter clarification | Short list of options or ask to select in the editor |
| **Partial or risky multi-file edit** | Batch touches ≥2 uris without `multiFileAllowed` on intent | Reject at validation | “That would change multiple files. Say ‘allow multi-file’ to continue or pick one file.” |
| **Model proposes edit outside resolved target** | Post-parse check: `replace_range` not ⊆ allowed span (derived from `ResolvedTarget`) | Reject batch | “That change was too broad for the selected scope.” |
| **Empty or whitespace-only model text** | `newText` trim empty on `replace_range` | Treat as no-op or abort (policy: abort for voice clarity) | “No change produced; say it another way.” |
| **Barge-in / cancel** | User interrupt or cancel intent | Kill in-flight LLM; apply nothing | “Cancelled.” |

**Safeguards summary**  
- **Fail closed** on validation errors.  
- **Never** apply partial batches when `atomic: true` (default).  
- **Voice-first recovery**: always offer a **next step** (retry, re-select, clarify, search again).
